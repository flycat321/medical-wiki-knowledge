  四、实施阶段

  Phase 1：向量化 + RAG 检索引擎（核心）

  目标： 让现有知识库支持语义搜索

  新增文件：
  epc-knowledge-base/
  ├── rag/
  │   ├── __init__.py
  │   ├── embeddings.py      # 调用通义千问嵌入API
  │   ├── vectorstore.py     # ChromaDB 管理
  │   ├── retriever.py       # 混合检索（向量+FTS5）
  │   └── chat.py            # 大模型对话+RAG组装

  关键逻辑：
  1. 文章保存时 → 自动切分段落 → 调用嵌入 API → 存入 ChromaDB
  2. 用户提问 → 同时走向量检索 + FTS5 关键词 → 合并去重 → Top-K 相关段落
  3. 将相关段落作为 context 注入 prompt → 调用 qwen3.5-plus → 返回答案+引用来源

  新增依赖：
  chromadb>=0.5.0
  duckduckgo-search>=6.0
  openai>=1.0          # 通义千问兼容 OpenAI SDK

  Phase 2：AI 对话界面

  目标： 提供一个对话式问答页面

  新增路由：
  GET  /chat                    # 对话页面
  POST /api/chat                # 对话API（SSE流式）
  GET  /api/chat/history        # 历史对话

  前端： 新增 templates/chat.html
  - 左侧：历史对话列表
  - 右侧：对话窗口（支持流式输出）
  - 每条回答底部：引用来源链接（点击跳转到对应文章）
  - 回答旁边：「提交审核」按钮（将联网搜索的结果提交为候选知识）

  ---
  Phase 3：联网搜索回退

  目标： 知识库无相关结果时，自动联网搜索

  逻辑：
  用户提问
    ↓
  RAG 检索（向量+FTS5）
    ↓
  相关度评分 > 阈值？
    ├── 是 → 基于知识库回答（标记：✅ 来自内部知识库）
    └── 否 → 触发联网搜索（DuckDuckGo）
              ↓
            搜索结果作为 context → 大模型回答
            标记：🌐 来自联网搜索（未经审核）
            显示「提交审核」按钮

  ---
  Phase 4：审核工作流

  目标： 管理员审核联网搜索产生的新知识

  数据库新增表：
  -- 待审核知识条目
  CREATE TABLE pending_knowledge (
      id INTEGER PRIMARY KEY,
      title TEXT NOT NULL,
      content TEXT NOT NULL,
      source_url TEXT,           -- 来源URL
      source_query TEXT,         -- 原始提问
      submitted_by TEXT,         -- 提交人
      status TEXT DEFAULT 'pending',  -- pending/approved/rejected
      reviewer TEXT,             -- 审核人
      review_note TEXT,          -- 审核备注
      category_id INTEGER,       -- 审核通过后归入的分类
      created TEXT,
      reviewed_at TEXT
  );

  管理员界面新增：
  GET  /admin/review              # 待审核列表
  GET  /admin/review/{id}         # 审核详情
  POST /admin/review/{id}/approve # 通过 → 自动创建文章+向量入库
  POST /admin/review/{id}/reject  # 拒绝

  流程：
  联网搜索回答 → 用户点击「提交审核」
    ↓
  进入 pending_knowledge 表（status=pending）
    ↓
  管理员在后台看到待审列表
    ↓
  审核：可编辑内容、选择分类
    ├── 通过 → 自动创建 article + 向量嵌入 + FTS5 索引
    └── 拒绝 → 标记原因，不入库

  ---
  Phase 5：多用户 + 经验提交

  目标： 团队成员可以直接提交经验

  数据库新增：
  -- 用户表（替代现有单一 admins 表）
  CREATE TABLE users (
      id INTEGER PRIMARY KEY,
      username TEXT UNIQUE NOT NULL,
      password TEXT NOT NULL,       -- bcrypt 哈希
      display_name TEXT,
      role TEXT DEFAULT 'member',   -- member/admin
      department TEXT,
      created TEXT
  );

  新增功能：
  - 普通用户登录后可提交经验（进入审核队列）
  - 管理员审核后发布
  - 用户可查看自己提交的条目状态
